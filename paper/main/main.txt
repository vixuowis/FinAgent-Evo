                                                  Abstract

 1            Existing financial agent benchmarks and systems predominantly evalu-
 2            ate isolated, single-skill capabilities—such as sentiment analysis or nu-
 3            merical reasoning—leaving end-to-end complex financial analysis largely
 4            unaddressed. Real-world financial workflows, however, require chaining
 5            heterogeneous tools across multiple steps, where skill dependencies, tool
 6            errors, and numerical hallucinations compound into brittle long-horizon
 7            failures. We present EvoFinAgent, a self-evolving multi-skill orchestration
 8            agent for complex financial analysis that treats each skill as an evolvable
 9            prompt genotype and distills episodic experiences into reusable procedural
10            rules. EvoFinAgent combines (1) an LLM-driven evolution engine that
11            mutates skill prompts using execution feedback, (2) a hierarchical memory
12            that abstracts cross-task heuristics, and (3) a DAG-style orchestrator that
13            executes structured tool plans with numerical consistency audit. To enable
14            rigorous end-to-end evaluation, we introduce a scalable N = 300 complex
15            multi-skill orchestration benchmark spanning 10 financial scenarios and three
16            difficulty tiers, with real-time API grounding and structured evaluation
17            criteria. EvoFinAgent achieves an overall judge score of 39.5, the highest
18            among all evaluated systems. Ablations confirm that each component con-
19            tributes meaningfully, and the benchmark design supports straightforward
20            extension to new scenarios and tool sets.


21   1    Introduction

22   LLM agents for finance must combine heterogeneous capabilities: retrieving real-time
23   prices, parsing financial statements, incorporating macro indicators, and performing precise
24   calculations. In practice, these tasks form a dependency graph: data must be fetched
25   before analysis, and all quantitative claims should be computed rather than hallucinated.
26   Unfortunately, agents with fixed prompts and static tool routing often degrade under longer
27   horizons, tool errors, and distribution shift, especially in finance where non-stationarity is
28   the norm.
29   We study the problem of robust multi-skill financial agents under complex orchestration.
30   Our core hypothesis is that robustness can be improved by enabling agents to self-improve
31   on two axes: evolving skill prompts using explicit feedback, and abstracting procedural rules
32   from past successful trajectories to guide future planning. We instantiate this hypothesis in
33   EvoFinAgent, a framework built around evolvable skills, hierarchical memory, and DAG
34   execution with verifiable tool use.

35   Problem setting. We consider financial tasks that require chaining multiple tools and
36   skills (e.g., retrieve → transform → compute → decide) under realistic failure modes: tool
37   latency, transient API errors, missing values, and distribution shift over time. We assume
38   that (i) intermediate tool outputs can be recorded and reused, and (ii) numerical claims
39   should be backed by programmatic computation whenever possible. In our setting, the agent
40   receives (a) a natural language query, (b) a fixed tool set with explicit input/output interfaces,
41   and (c) task-level evaluation criteria specifying required answer metrics. The agent must
42   output a structured final response that covers all required metrics, with quantitative claims
43   either computed via a Python interpreter or directly traceable to tool outputs. We assume
44   transient tool failures can be handled by bounded retries and that cached tool responses can
45   be reused to control non-determinism.

46   Contributions. EvoFinAgent targets complex orchestration scenarios while still sup-
47   porting standard financial reasoning benchmarks. Our main contributions are:

48        • Evolvable skill library. We formalize each skill as a prompt genotype with
49          interface constraints and propose an LLM-driven mutation operator that updates
50          skills using execution feedback, without changing the orchestrator logic.
51        • Hierarchical memory → procedural rules. We introduce a three-layer memory
52          design and a rule abstraction mechanism that distills episodic trajectories into
53          compact, reusable procedural constraints to improve planning reliability.
54        • Verified DAG execution. We propose a structured, tool-grounded execution
55          protocol that represents plans as typed JSON steps, executes them in a DAG-style
56          dependency order, and enforces programmatic computation for quantitative claims.
57        • Empirical study. We evaluate on a complex multi-skill orchestration benchmark
58          with real-time financial APIs and LLM-as-a-judge scoring, and audit baseline parity
59          and failure modes. We provide a paper-ready benchmark card (taxonomy, task
60          counts, task examples, tool versions) and a fully specified judge protocol (model,
61          temperature, exact prompt text, output schema).

62   Research questions. We focus on three questions: (1) Can prompt-level evolution improve
63   skill specialization with minimal supervision? (2) Can hierarchical memory distill reusable
64   heuristics that make orchestration more reliable? (3) Does the combined system outperform
65   ReAct-style baselines on multi-tool financial workflows under realistic tool noise?
66   As illustrated in Figure 1, standard auto-regressive agents (left) often struggle with such
67   complex workflows, falling into redundant loops or losing track of critical variables like
68   exchange rates. In contrast, our proposed EvoFinAgent (right) decomposes the task into
69   a structured Directed Acyclic Graph (DAG), ensuring clear data dependencies, persistent
70   memory states, and verified execution at each step.

109   3     EvoFinAgent
110   Overview. Given a task instruction x (natural language) and a tool set T (e.g., quote
111   API, FX API, news search, Python interpreter), the goal is to produce an answer y that
112   maximizes a correctness and grounding reward R(x, y, T ). Formally, we seek a policy π that
113   generates a structured plan P and yields y:
                                  π ∗ = arg max Ex∼D [R(x, π(x, T ), T )]                         (1)
                                             π

114   EvoFinAgent maintains a skill library S, where each skill s ∈ S is an LLM-callable module
115   with an explicit interface contract. At test time, the orchestrator policy generates a plan that
116   composes skills into a Directed Acyclic Graph (DAG), executes the plan with verification,
117   and logs trajectories for self-improvement. Across episodes, the evolution engine mutates
118   skills based on feedback, while the memory module abstracts successful experience into
119   procedural rules injected into future planning.

120   3.1   Skill Genotypes

121   Each skill s is represented as a genotype g(s) comprising: a unique identifier, a backbone LLM
122   configuration (model, temperature, context budget), a prompt chromosome (parameterized
123   system prompt), a Pydantic-based interface contract (strict input/output schema), and
124   dependency metadata (required upstream artifacts and tool access). A genotype is compiled
125   into an executable tool by instantiating its prompt chromosome, attaching the interface
126   contract as a validator, and invoking the backbone LLM. Enforcing explicit schemas prevents
127   silent format drift—a common failure mode in multi-tool pipelines where a downstream step
128   expects a number but receives narrative text.

129   3.2   LLM-Driven Evolution Engine

130   Each episode produces a trace τ (plan, tool calls, intermediate artifacts, final output) and a
131   feedback signal F (τ ) (judge score, constraint violations, success/failure tags). The mutation

134   from τ , (ii) propose targeted edits to θs (additional guardrails, few-shot examples, tighter

136   a library of up to 50 skills per category; skills on critical failure paths are prioritized for
137   mutation. A candidate s′ replaces s only if it passes a replay buffer of 10 characteristic tasks,
138   preventing regressions.

139   3.3   Hierarchical Memory and Procedural Rules
140   EvoFinAgent maintains three layers: working memory (current episode), episodic
141   memory (past trajectories scored by tool-error frequency, recovery complexity, and novelty;
142   stored when I(τ ) ≥ Tmem = 0.7), and procedural memory (abstracted rules). When
143   episodic failures reach critical mass, a meta-model distills them into procedural rules R =
144   A(E), each a ⟨condition, constraint, rationale⟩ tuple. At inference, a relevant subset Rx
145   is retrieved and injected into planning: P ∼ π(· | x, T , Rx ), functioning as soft-to-hard
146   guardrails that reduce recurring orchestration errors.

147   3.4   Verified Multi-Skill Orchestration

152   loss, we enforce static type safety before execution. Each node v ∈ V (a skill execution) is
153   defined by a signature fv : Iv → Ov , where Iv and Ov are input and output schema spaces.
154   A generated DAG G is deemed valid if and only if it satisfies the type-matching constraint

173   Error handling. When a step fails (API timeout, schema mismatch, exception), the
174   executor applies a recovery policy: retry with backoff, fall back to alternative tools/skills, or
175   re-plan with updated constraints. Failures and recoveries are written to episodic memory
176   and become training signals for evolution.
177   The system supports real-time data retrieval via financial APIs (stock quotes, FX rates,
178   macro indicators) and can discover additional remote tools when internal capabilities are
179   insufficient.

180   4     Experimental Setup
181   4.1   Benchmark

182   We evaluate on N = 300 complex financial tasks spanning 10 scenarios (Equity Research,
183   Macro Strategy, Portfolio Risk, Crypto Assets, Event-Driven, Fixed Income, Commodi-
184   ties, ESG, Quantitative Factors, Global Banking), each with 10 Easy (1–2 tools), 10
185   Medium (3–4 tools), and 10 Hard (≥5 tools) tasks. Tasks require chaining real-time
186   data retrieval (stock/crypto prices, FX, macro indicators, financial statements, news
187   search via QVeris MCP) and programmatic computation (Python interpreter). Each
188   task specifies evaluation_criteria with final_answer_metrics, min_tool_calls, and
189   must_call_sequence. All methods share the same tool set with pinned versions; cache-at-
190   time controls non-determinism.

191   4.2   Baselines and Ablations

192   ReAct [Yao et al., 2023b]: standard interleaved reasoning-action agent with the same tool
193   set, capped at 15 steps to prevent unbounded loops. FinAgent [Li et al., 2024]: ReAct-style
194   financial agent with flat single-pass prompting, no structured planning or skill specialization.
195   FinMem [Xie et al., 2023]: ReAct augmented with a layered memory module for trading
196   decisions. All baselines share identical tool interfaces, pinned tool versions, and runtime
197   budgets (180s timeout, 3 retries).
198   Ablations isolate each component: w/o Evolution (skill prompts fixed, no feedback-driven
199   mutation), w/o Memory (planning without procedural rule injection), w/o Orchestration
200   (zero-shot ReAct pattern, no DAG execution or verification).

201   4.3   Evaluation Protocol

202   We report two complementary metrics: (i) Hard-success rate — fraction of tasks com-
203   pleting without timeout (180s) or terminal exception, measuring execution robustness; (ii)
204   Judge score — LLM-as-a-judge on a 0–100 scale evaluating completeness, tool grounding,
205   orchestration quality, and clarity. The judge ensemble uses GPT-5.4 and Claude-Opus-4-6
206   at temperature 0; we report mean and 95% CI across all N = 300 tasks.

207   5     Results
208   5.1   Complex Multi-Skill Orchestration

209   On the N = 300 complex financial tasks across 10 scenarios, EvoFinAgent achieves an
210   overall judge score of 39.5 (± 2.6) and a hard-success rate of 74.4%, outperforming the
211   FinAgent baseline in judge score (28.4, 71.3%) and FinMem (24.2, 76.2%). Note that FinMem
212   achieves a slightly higher hard-success rate (76.2%) due to its shallower execution depth, but
213   at the cost of significantly lower reasoning quality.

217   S10: Multi-Source Grounding). The performance gap narrows slightly in S5 (FX Risk &
218   Hedging), where shorter tool chains reduce the advantage of structured DAG execution.

254   5.2                  Analysis of Hard Success Rates
255   While EvoFinAgent achieves the highest Judge Score across tasks, its Hard Success rate
256   (74.4%) is occasionally lower than simpler baselines like FinMem (76.2%) on specific scenarios.
257   Figure 4 illustrates this breakdown across task complexity (defined by the number of required
258   tools). This reflects an inherent trade-off in multi-tool orchestration: attempting deeper,
259   multi-step DAG plans (averaging 6.2 steps) naturally increases the exposure to API timeouts,
260   context limits, and REPL variable drops compared to shallow execution (1.5 steps). However,
261   the baselines’ seemingly higher completion rates are often “hollow,” resulting in fabricated
262   or ungrounded answers, as evidenced by their significantly lower Judge Scores.

287   5.4   Compute Resources and API Costs
288   All agent experiments were conducted on a single server equipped with an AMD EPYC CPU,
289   128GB RAM, and 1× NVIDIA A100 GPU (primarily used for local embedding generation
290   and parsing tools). The core reasoning and orchestration engines relied on external APIs
291   (GPT-4o, GPT-4o-mini, and Qwen2.5).
292   For the main N = 300 complex orchestration benchmark (including all ablations and
293   baselines), the total estimated API cost was approximately $450 USD. The execution time
294   for a single complex task averaged 45 seconds under the EvoFinAgent DAG orchestrator,
295   compared to 120+ seconds for the ReAct baseline due to redundant looping. The full
296   benchmark evaluation suite took roughly 48 hours of wall-clock time to complete.

297   6     Limitations and Future Work
298   Semantic labeling. EvoFinAgent excels at numerical reasoning but shows a gap on fine-
299   grained semantic labeling in dense financial text (e.g., SEC filings); integrating specialized
300   NLP parsers is a natural next step.
301   REPL state persistence. The python_interpreter skill occasionally loses variable scope
302   across steps in long-chain computations; a unified cross-skill execution sandbox would address
303   this.
304   Evolutionary lag. The evolution engine operates on episodic batches, introducing a
305   transient vulnerability window when APIs change field names abruptly.
306   Reproducibility. Real-time APIs introduce non-determinism; we use cache-at-time and
307   log all tool outputs. Human evaluation on a subset confirms judge reliability (Pearson
308   r = 0.945).
309   Safety. This work targets technical robustness; deployment for automated trading requires
310   additional compliance safeguards.

311   7     Conclusion
312   EvoFinAgent demonstrates that combining evolvable skill prompts, hierarchical memory
313   abstraction, and verified DAG execution can significantly improve multi-skill financial
314   orchestration. We hope this work motivates more rigorous, tool-grounded evaluation of
315   self-improving agents in non-stationary domains such as finance.

316   Future Work. Future research will focus on (i) scaling the evolution engine to thousands
317   of concurrent skills, and (ii) integrating a more diverse set of multi-modal tools, including

319   Ethics Statement
320   This work develops an AI agent framework for financial analysis and research assistance. All

524   A.6    LLM-as-a-judge rubric

525   We score each answer on a 0–100 scale as a weighted sum:

526         • Completeness (0–40): all required fields present; key metrics included.
527         • Tool grounding (0–35): claims are supported by logged tool outputs; numbers
528           are computed.
529         • Orchestration quality (0–15): correct sequencing; handles missing values; no
530           unnecessary calls.
531         • Clarity (0–10): concise, well-structured, and actionable recommendation.

532   A.7    Judge prompt (exact)

533   We use a fixed rubric prompt and deterministic decoding (temperature 0). The judge returns
534   a JSON object with fields {score, reasoning, met_metrics, missed_metrics}.
535
536   You are a strict financial evaluation judge (LLM-as-a-Judge). You will be given:
537   1) Task Query
538   2) evaluation_criteria (containing final_answer_metrics / min_tool_calls / must_call_sequence)
539   3) Agent’s Final Answer
540   4) Agent’s trajectory (sequence of tool calls, possibly including parameters and output summaries)
541
542   Please provide a comprehensive score from 0-100 and explain the reasoning, focusing on:
543   - Whether the final answer covers the key points of final_answer_metrics (clearly state met/missing items)
544   - Whether the trajectory reflects reasonable data acquisition and calculation (avoid "fabrication")
545   - Whether it satisfies the intent of min_tool_calls (quantity is for reference; focus on necessary steps)
546   - must_call_sequence is for "intent sequence" reference: if tool names don’t match exactly, judge if the
547        ,→ intent aligns (e.g., News Search -> Data Fetch -> Calculation -> Synthesis)
