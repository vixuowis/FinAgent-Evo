Real-world financial workflows require chaining heterogeneous tools across multiple steps, where skill dependencies, tool errors, and numerical hallucinations compound into brittle long-horizon failures.
However, existing financial agent benchmarks and systems predominantly evaluate isolated, single-skill capabilities (e.g, sentiment analysis, numerical reasoning), leaving end-to-end complex financial analysis largely unaddressed. 
To this end, we present \textsc{EvoFinAgent}, a self-evolving multi-skill orchestration agent for complex financial analysis that treats each skill as an evolvable skill prompt (i.e., \emph{genotype}) and distills episodic experiences into reusable \emph{procedural rules}.
In particular, \textsc{EvoFinAgent} introduces (1) an LLM-driven evolution engine that mutates skill prompts using execution feedback, (2) a hierarchical memory that abstracts cross-task heuristics, and (3) a DAG-style orchestrator that executes structured tool plans with \emph{numerical consistency audit}.
To enable rigorous end-to-end evaluation, we introduce a scalable benchmark of 300 complex multi-skill orchestration tasks, spanning 10 financial scenarios and three difficulty tiers, with real-time API grounding and structured evaluation criteria. 
\textsc{EvoFinAgent} achieves an overall judge score of 39.5, the highest among all evaluated systems. Ablations confirm that each component contributes meaningfully, and the benchmark design supports straightforward extension to new scenarios and tool sets.

\end{abstract}

\section{Introduction}

\eat{LLM agents for finance must combine heterogeneous capabilities: retrieving real-time prices, parsing financial statements, incorporating macro indicators, and performing precise calculations. In practice, these tasks form a dependency graph: data must be fetched before analysis, and all quantitative claims should be computed rather than hallucinated. Unfortunately, agents with fixed prompts and static tool routing often degrade under longer horizons, tool errors, and distribution shift, especially in finance where non-stationarity is the norm.}

LLM agents for financial applications need to orchestrate a heterogeneous set of capabilities, including retrieving real-time market prices, parsing structured and unstructured financial statements, integrating macroeconomic indicators, and executing precise numerical computations.
In practice, these subtasks form a directed dependency graph in which data acquisition precedes downstream analysis, and quantitative claims should be derived through explicit computation rather than generated from the model's parametric memory.
Unfortunately, agents with fixed prompts and static tool-routing policies tend to degrade over long horizons, under tool failures, and across distribution shifts, particularly in finance where non-stationarity is the norm.

We study the problem of building robust multi-skill financial agents that operate under complex orchestration. 
Our key hypothesis is that robustness can be improved by enabling agents to \emph{self-improve} along two complementary axes: evolving skill prompts through explicit feedback, and abstracting procedural rules from past successful trajectories to guide future planning. 
We instantiate this hypothesis in \textsc{EvoFinAgent}, a framework built around evolvable skills, hierarchical memory, and Directed Acyclic Graph (DAG) execution with verifiable tool use.

\paragraph{Problem Setting.}
We consider financial tasks that require chaining multiple tools and skills (e.g., \emph{retrieve} $\rightarrow$ \emph{transform} $\rightarrow$ \emph{compute} $\rightarrow$ \emph{decide}) under realistic failure modes: tool latency, transient API errors, missing values, and distribution shift over time. 
Our setting rests on two assumptions: (i) intermediate tool outputs can be recorded and reused, and (ii) numerical claims should be backed by programmatic computation whenever possible.
In our setting, the agent receives (a) a natural language query, (b) a fixed tool set with explicit input/output interfaces, and (c) task-level evaluation criteria specifying required answer metrics. 
The agent is expected to produce a structured final response that covers all required metrics, with quantitative claims either computed via a Python interpreter or directly traceable to the tool's outputs. 
We assume that transient tool failures can be handled by bounded retries and that cached tool responses can be reused to control non-determinism.

\paragraph{Contributions.}
\textsc{EvoFinAgent} targets complex orchestration scenarios while still supporting standard financial reasoning benchmarks. Our main contributions are:
\begin{itemize}
  \item \textbf{Evolvable skill library.} We formalize each skill as a prompt genotype with interface constraints and propose an LLM-driven mutation operator that updates skills using execution feedback, without changing the orchestrator logic.
  \item \textbf{Hierarchical memory $\rightarrow$ procedural rules.} We introduce a three-layer memory design and a rule abstraction mechanism that distills episodic trajectories into compact, reusable procedural constraints to improve planning reliability.
  \item \textbf{Verified DAG execution.} We propose a structured, tool-grounded execution protocol that represents plans as typed JSON steps, executes them in a DAG-style dependency order, and enforces programmatic computation for quantitative claims.
  \item \textbf{Empirical study.} We evaluate on a complex multi-skill orchestration benchmark with real-time financial APIs and LLM-as-a-judge scoring, and audit baseline parity and failure modes. We provide a paper-ready benchmark card (taxonomy, task counts, task examples, tool versions) and a fully specified judge protocol (model, temperature, exact prompt text, output schema).
\end{itemize}

\paragraph{Research Questions.}
We focus on three questions: (1) Can prompt-level evolution improve skill specialization with minimal supervision? (2) Can hierarchical memory distill reusable heuristics that make orchestration more reliable? (3) Does the combined system outperform ReAct-style baselines on multi-tool financial workflows under realistic tool noise?

As illustrated in Figure~\ref{fig:dag_comparison}, standard auto-regressive agents (left) often struggle with such complex workflows, falling into redundant loops or losing track of critical variables like exchange rates. In contrast, our proposed \textsc{EvoFinAgent} (right) decomposes the task into a structured DAG, ensuring clear data dependencies, persistent memory states, and verified execution at each step.

\section{Related Work}
\paragraph{Tool-augmented agents.}
ReAct~\citep{yao2023react} interleaves reasoning and actions with environment feedback; ReWOO~\citep{xu2023rewoo} pre-generates full tool-call plans to reduce redundant LLM calls. Toolformer~\citep{schick2023toolformer} and HuggingGPT~\citep{shen2023hugginggpt} scale tool usage and routing, while Gorilla~\citep{patil2024gorilla} mitigates API hallucination via retrieval. MetaGPT~\citep{hong2023metagpt} enables structured multi-agent collaboration. Our work targets robust financial workflows where tool dependencies are structured and quantitative claims must be verifiable.

\paragraph{Self-improvement and memory.}
Reflexion~\citep{shinn2023reflexion} and Self-Refine~\citep{madaan2023selfrefine} improve agents via linguistic feedback without fine-tuning. Voyager~\citep{wang2023voyager} maintains a growing skill library refined across tasks—the closest parallel to our evolvable skill library. Generative Agents~\citep{park2023generativeagents} and RAG~\citep{lewis2020rag} demonstrate memory-reflection-planning and retrieval-grounded generation; FinMem~\citep{finmem2023} applies layered memory to trading. We extend this by distilling episodic trajectories into \emph{procedural rules} injected into planning.

\paragraph{Verified computation and benchmarks.}
Numerical fidelity is a persistent failure mode for LLM agents. PAL~\citep{gao2022pal} and PoT~\citep{chen2022pot} offload arithmetic to an external interpreter by generating executable programs as intermediate reasoning; we enforce this strictly for all quantitative claims via a Python REPL with schema-validated outputs. Self-consistency~\citep{wang2023selfconsistency} further improves robustness by aggregating diverse reasoning paths. For evaluation, interactive environments such as ALFWorld~\citep{shridhar2021alfworld} and WebShop~\citep{yao2022webshop} provide grounded sequential decision-making benchmarks; AgentBench~\citep{liu2023agentbench} and API-Bank~\citep{li2023apibank} evaluate tool-using agents across diverse environments. Our benchmark complements these by targeting complex multi-tool financial workflows with real-time APIs and structured verification.

\paragraph{Financial agents and prompt evolution.}
Domain-adapted financial LLMs such as BloombergGPT~\citep{wu2023bloomberggpt} and FinGPT~\citep{yang2023fingpt} demonstrate the value of finance-specific pretraining. Prior multi-agent systems explore tool-grounded financial decision making~\citep{tradingagents2024,fincon2024} and memory-centric trading~\citep{finmem2023}. For prompt optimization, EvoPrompt~\citep{evoprompt2024} and OPRO~\citep{opro2023} use LLMs as crossover and mutation operators to evolve natural language prompts; we adopt this evolutionary perspective but apply it to a \emph{skill genotype} search space—where the genotype is a skill prompt with interface constraints—guided by execution feedback rather than held-out accuracy.

\section{EvoFinAgent}
\paragraph{Overview.}
Given a task instruction $x$ (natural language) and a tool set $\mathcal{T}$ (e.g., quote API, FX API, news search, Python interpreter), the goal is to produce an answer $y$ that maximizes a correctness and grounding reward $R(x, y, \mathcal{T})$. Formally, we seek a policy $\pi$ that generates a structured plan $P$ and yields $y$:
\begin{equation}
\pi^* = \arg\max_{\pi} \mathbb{E}_{x \sim \mathcal{D}} [ R(x, \pi(x, \mathcal{T}), \mathcal{T}) ]
\end{equation}
\textsc{EvoFinAgent} maintains a skill library $\mathcal{S}$, where each skill $s \in \mathcal{S}$ is an LLM-callable module with an explicit \emph{interface contract}. At test time, the orchestrator policy generates a plan that composes skills into a Directed Acyclic Graph (DAG), executes the plan with verification, and logs trajectories for self-improvement. Across episodes, the evolution engine mutates skills based on feedback, while the memory module abstracts successful experience into procedural rules injected into future planning.

\subsection{Skill Genotypes}
Each skill $s$ is represented as a \emph{genotype} $g(s)$ comprising: a unique identifier, a backbone LLM configuration (model, temperature, context budget), a \emph{prompt chromosome} (parameterized system prompt), a Pydantic-based \emph{interface contract} (strict input/output schema), and dependency metadata (required upstream artifacts and tool access). A genotype is compiled into an executable tool by instantiating its prompt chromosome, attaching the interface contract as a validator, and invoking the backbone LLM. Enforcing explicit schemas prevents silent format drift—a common failure mode in multi-tool pipelines where a downstream step expects a number but receives narrative text.

\subsection{LLM-Driven Evolution Engine}
Each episode produces a trace $\tau$ (plan, tool calls, intermediate artifacts, final output) and a feedback signal $F(\tau)$ (judge score, constraint violations, success/failure tags). The mutation operator $M$ updates the prompt chromosome without gradient-based fine-tuning:
\begin{equation}
\theta_s^{(t+1)} \sim M\big(\theta_s^{(t)}, \tau, F(\tau)\big)
\end{equation}
Concretely, $M$ is prompted to (i) diagnose failure modes (format errors, hallucinated numbers) from $\tau$, (ii) propose targeted edits to $\theta_s^{(t)}$ (additional guardrails, few-shot examples, tighter interface constraints), and (iii) output the patched genotype in canonical JSON. We maintain a library of up to 50 skills per category; skills on critical failure paths are prioritized for mutation. A candidate $s'$ replaces $s$ only if it passes a replay buffer of 10 characteristic tasks, preventing regressions.

\subsection{Hierarchical Memory and Procedural Rules}
\textsc{EvoFinAgent} maintains three layers: \textbf{working memory} (current episode), \textbf{episodic memory} (past trajectories scored by tool-error frequency, recovery complexity, and novelty; stored when $I(\tau) \ge T_{\text{mem}}=0.7$), and \textbf{procedural memory} (abstracted rules). When episodic failures reach critical mass, a meta-model distills them into procedural rules $\mathcal{R} = A(\mathcal{E})$, each a $\langle\text{condition}, \text{constraint}, \text{rationale}\rangle$ tuple. At inference, a relevant subset $\mathcal{R}_x$ is retrieved and injected into planning: $P \sim \pi(\cdot \mid x, \mathcal{T}, \mathcal{R}_x)$, functioning as soft-to-hard guardrails that reduce recurring orchestration errors.

\subsection{Verified Multi-Skill Orchestration}
Given a task $x$, the orchestrator produces a structured plan represented as a Directed Acyclic Graph $G = (V, E)$. To ensure execution robustness, we formulate the planning phase as a \emph{strongly-typed probabilistic inference} problem.

\paragraph{Strongly-typed DAG generation.}
Unlike ReAct agents that suffer from runtime variable loss, we enforce static type safety before execution. Each node $v \in V$ (a skill execution) is defined by a signature $f_v: \mathcal{I}_v \rightarrow \mathcal{O}_v$, where $\mathcal{I}_v$ and $\mathcal{O}_v$ are input and output schema spaces. A generated DAG $G$ is deemed valid if and only if it satisfies the \emph{type-matching constraint} for all nodes:
\begin{equation}
\forall v \in V, \quad \mathcal{I}_v \subseteq x \cup \bigcup_{u \in \text{Pred}(v)} \mathcal{O}_u
\end{equation}
where $\text{Pred}(v)$ denotes the predecessor nodes of $v$. This static analysis guarantees that no skill is executed with missing dependencies, fundamentally preventing the ``logical entropy'' common in auto-regressive agents.

\paragraph{Rule-constrained probabilistic planning.}
The generation of the DAG is conditioned on the retrieved procedural rules $\mathcal{R}_x$. We model this as maximizing a constrained joint probability:
\begin{equation}
G^* = \arg\max_{G \in \mathcal{G}_{\text{valid}}} \prod_{v \in V} \phi_{\text{skill}}(v \mid x) \prod_{r \in \mathcal{R}_x} \psi_{\text{rule}}(G, r)
\end{equation}
where $\phi_{\text{skill}}$ denotes the LLM's prior probability of selecting a skill for the task, and $\psi_{\text{rule}}(G, r) \in [0, 1]$ is a potential function that heavily penalizes plans violating a procedural rule $r$ (e.g., retrieving a stock price without first querying its ticker).

\paragraph{Execution and verification.}
The executor maintains a state $S_t$ of verified artifacts. A node $v$ becomes executable when all its dependencies are resolved: $\{u \mid (u, v) \in E\} \subseteq V_{\text{completed}}$. Upon executing $v$ to produce output $o_v$, a strict verification function $V(o_v)$ is applied before adding it to $S_t$:
\begin{equation}
S_{t+1} = 
\begin{cases} 
S_t \cup \{o_v\} & \text{if } V(o_v) = \text{True} \\
S_t & \text{otherwise (trigger recovery)}
\end{cases}
\end{equation}
The verification $V(o_v)$ consists of:
(i) Pydantic schema validation of skill outputs,
(ii) numeric cross-check via Python REPL (e.g., verifying if the growth rate implies the target price),
and (iii) citation binding where each factual claim is linked to a unique tool output ID.
Any output that fails verification is either flagged for correction or triggers a local re-plan.

\paragraph{Error handling.}
When a step fails (API timeout, schema mismatch, exception), the executor applies a recovery policy: retry with backoff, fall back to alternative tools/skills, or re-plan with updated constraints. Failures and recoveries are written to episodic memory and become training signals for evolution.

The system supports real-time data retrieval via various financial APIs (e.g., stock quotes, FX rates, macro indicators), and can discover additional remote tools when internal capabilities are insufficient.

\section{Experiments}
\subsection{Benchmark}
We evaluate on $N=300$ complex financial tasks spanning 10 scenarios (Equity Research, Macro Strategy, Portfolio Risk, Crypto Assets, Event-Driven, Fixed Income, Commodities, ESG, Quantitative Factors, Global Banking), each with 10 Easy (1--2 tools), 10 Medium (3--4 tools), and 10 Hard ($\ge$5 tools) tasks. Tasks require chaining real-time data retrieval (stock/crypto prices, FX, macro indicators, financial statements, news search via QVeris MCP) and programmatic computation (Python interpreter). Each task specifies \texttt{evaluation\_criteria} with \texttt{final\_answer\_metrics}, \texttt{min\_tool\_calls}, and \texttt{must\_call\_sequence}. All methods share the same tool set with pinned versions; cache-at-time controls non-determinism.

\subsection{Baselines and Ablations}
\textbf{ReAct}~\citep{yao2023react}: standard interleaved reasoning-action agent with the same tool set, capped at 15 steps to prevent unbounded loops. \textbf{FinAgent}~\citep{tradingagents2024}: ReAct-style financial agent with flat single-pass prompting, no structured planning or skill specialization. \textbf{FinMem}~\citep{finmem2023}: ReAct augmented with a layered memory module for trading decisions. All baselines share identical tool interfaces, pinned tool versions, and runtime budgets (180s timeout, 3 retries).

Ablations isolate each component: \emph{w/o Evolution} (skill prompts fixed, no feedback-driven mutation), \emph{w/o Memory} (planning without procedural rule injection), \emph{w/o Orchestration} (zero-shot ReAct pattern, no DAG execution or verification).

\subsection{Evaluation Protocol}
We report two complementary metrics: (i) \textbf{Hard-success rate} — fraction of tasks completing without timeout (180s) or terminal exception, measuring execution robustness; (ii) \textbf{Judge score} — LLM-as-a-judge on a 0--100 scale evaluating completeness, tool grounding, orchestration quality, and clarity. The judge ensemble uses GPT-5.4 and Claude-Opus-4-6 at temperature 0; we report mean and 95\% CI across all $N=300$ tasks.

\subsection{Main Results}
On the $N=300$ complex financial tasks across 10 scenarios, \textsc{EvoFinAgent} achieves an overall judge score of 39.5 ($\pm$ 2.6) and a hard-success rate of 74.4\%, outperforming the FinAgent baseline in judge score (28.4, 71.3\%) and FinMem (24.2, 76.2\%). Note that FinMem achieves a slightly higher hard-success rate (76.2\%) due to its shallower execution depth, but at the cost of significantly lower reasoning quality.

\paragraph{Impact of Evolution Engine.}
The performance gap between the Full system and the \texttt{w/o Evolution} variant (39.5 vs.\ 36.5) highlights the efficacy of our feedback-driven mutation operator. Approximately 26\% of complex tasks were ``salvaged'' by mutations that tightened interface contracts or added grounding instructions after initial failures. The meta-model accepted mutations at an estimated rate of 15--20\%, demonstrating that even infrequent prompt-level refinements can cumulatively resolve structural bottlenecks in multi-skill orchestration.

\paragraph{Impact of Hierarchical Memory.}
Removing procedural rule injection (\texttt{w/o Memory}) reduces the judge score from 36.5 to 34.9. 
The drop is most pronounced on Hard tasks ($\ge$5 tools), where recurring failure patterns—such as missing fallback logic for API timeouts—are most likely to recur across episodes. 
Procedural rules can effectively encode these patterns as reusable constraints, reducing the planner's reliance on in-context reasoning for known failure modes.

\paragraph{Key Insight.}
The substantial judge score gap between the Full system (39.5) and the ReAct baseline (4.1) is primarily driven by \textbf{workflow control}. Without structured orchestration, agents lose track of intermediate state and make contradictory tool calls in financial tasks requiring more than 4 steps, even when using the same backbone model. Note that on Scenario 5 (Event-Driven \& Policy Analysis), the FinAgent baseline outperforms \textsc{EvoFinAgent} (36.9 vs.\ 30.4), suggesting that event-driven tasks with shorter tool chains benefit less from DAG overhead and more from direct retrieval-and-respond patterns.

The DAG orchestrator's structured planning reduces redundant tool calls, keeping costs predictable even as task complexity scales.

\subsection{Analysis of Hard Success Rates}
While \textsc{EvoFinAgent} achieves the highest Judge Score across tasks, its Hard Success rate (74.4\%) is occasionally lower than simpler baselines like FinMem (76.2\%) on specific scenarios. Figure~\ref{fig:hard_success} illustrates this breakdown across task complexity (defined by the number of required tools). This reflects an inherent trade-off in multi-tool orchestration: attempting deeper, multi-step DAG plans (averaging 6.2 steps) naturally increases the exposure to API timeouts, context limits, and REPL variable drops compared to shallow execution (1.5 steps). However, the baselines' seemingly higher completion rates are often ``hollow,'' resulting in fabricated or ungrounded answers, as evidenced by their significantly lower Judge Scores.



\subsection{Compute Resources and API Costs}
All agent experiments were conducted on a single server equipped with an AMD EPYC CPU, 128GB RAM, and 1$\times$ NVIDIA A100 GPU (primarily used for local embedding generation and parsing tools).
The core reasoning and orchestration engines relied on external APIs (GPT-4o, GPT-4o-mini, and Qwen2.5).

For the main $N=300$ complex orchestration benchmark (including all ablations and baselines), the total estimated API cost was approximately \$450 USD.
The execution time for a single complex task averaged 45 seconds under the \textsc{EvoFinAgent} DAG orchestrator, compared to 120+ seconds for the ReAct baseline due to redundant looping. The full benchmark evaluation suite took roughly 48 hours of wall-clock time to complete.


\section{Limitations and Future Work}
\label{sec:limitations}

\textbf{Semantic labeling.} \textsc{EvoFinAgent} excels at numerical reasoning but shows a gap on fine-grained semantic labeling in dense financial text (e.g., SEC filings); integrating specialized NLP parsers is a natural next step.

\textbf{REPL state persistence.} The \texttt{python\_interpreter} skill occasionally loses variable scope across steps in long-chain computations; A unified cross-skill execution sandbox would address this issue.

\textbf{Evolutionary lag.} The evolution engine operates on episodic batches, introducing a transient vulnerability window when APIs change field names abruptly.

\textbf{Reproducibility.} Real-time APIs introduce non-determinism; we use cache-at-time and log all tool outputs. Human evaluation on a subset confirms judge reliability (Pearson $r=0.945$).

\textbf{Safety.} This work targets technical robustness; deployment for automated trading requires additional compliance safeguards.

\paragraph{Future Work.}
Future research will focus on (i) scaling the evolution engine to thousands of concurrent skills, and (ii) integrating a more diverse set of multi-modal tools, including OCR for PDF analysis and chart-to-text generation for technical analysis.

\section{Conclusion}
\textsc{EvoFinAgent} demonstrates that combining evolvable skill prompts, hierarchical memory abstraction, and verified DAG execution can significantly improve multi-skill financial orchestration. We hope this work motivates more rigorous, tool-grounded evaluation of self-improving agents in non-stationary domains such as finance.

\subsection{LLM-as-a-judge rubric}
We score each answer on a 0--100 scale as a weighted sum:
\begin{itemize}
  \item \textbf{Completeness (0--40):} all required fields present; key metrics included.
  \item \textbf{Tool grounding (0--35):} claims are supported by logged tool outputs; numbers are computed.
  \item \textbf{Orchestration quality (0--15):} correct sequencing; handles missing values; no unnecessary calls.
  \item \textbf{Clarity (0--10):} concise, well-structured, and actionable recommendation.
\end{itemize}

\subsection{Judge prompt (exact)}
We use a fixed rubric prompt and deterministic decoding (temperature 0). The judge returns a JSON object with fields \texttt{\{score, reasoning, met\_metrics, missed\_metrics\}}.

\begin{lstlisting}
You are a strict financial evaluation judge (LLM-as-a-Judge). You will be given:
1) Task Query
2) evaluation_criteria (containing final_answer_metrics / min_tool_calls / must_call_sequence)
3) Agent's Final Answer
4) Agent's trajectory (sequence of tool calls, possibly including parameters and output summaries)

Please provide a comprehensive score from 0-100 and explain the reasoning, focusing on:
- Whether the final answer covers the key points of final_answer_metrics (clearly state met/missing items)
- Whether the trajectory reflects reasonable data acquisition and calculation (avoid "fabrication")
- Whether it satisfies the intent of min_tool_calls (quantity is for reference; focus on necessary steps)
- must_call_sequence is for "intent sequence" reference: if tool names don't match exactly, judge if the intent aligns (e.g., News Search -> Data Fetch -> Calculation -> Synthesis)

  "reasoning": "",

  "missed_metrics": []
}
